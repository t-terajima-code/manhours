対象者: 工数集計担当者、経理部門

はじめに

このマニュアルは、毎月の工数集計・按分処理を実行する方向けのガイドです。技術的な詳細は含まず、実行手順と確認方法に重点を置いています。

重要: このシステムは 2段階処理 を採用しており、月初と月中旬～下旬に分けて実行します。

第1章 システム概要（5分で理解）

システムの役割

工数集計・按分システムは、従業員が入力した工数（作業時間）を自動的に集計し、経理チームから提供される損益Excelに基づいて案件ごとにコストを計算するシステムです。

実務における 2 段階処理

このシステムは、実務のスケジュールに合わせて 2つのバッチを使い分けます。

        
            【月初】稼働5日以内
              └─ run_process.bat 実行
                 → 工数集計～検証～配分
                 → 工数エラーを月初に検出・修正
                 
            【15日～25日ごろ】損益Excel到着後
              └─ run_allocate_costs.bat 実行
                 → コスト按分（最新損益データ反映）
                 → 経理データを出力・提出

なぜ 2 段階か？

フェーズ 実行時期 実行ファイル 処理内容 所要時間
Stage 1 稼働5日以内 run_process.bat 工数集計～検証～配分 約5分
Stage 2 損益Excel到着時 run_allocate_costs.bat コスト按分 約2分
（実務：15日～25日ごろ） （最新損益データ反映）

✅ Stage 1（稼働5日以内）: 月初の早期段階で実施

  • 工数集計
  • 勤怠検証（エラーチェック）
  • 実績配分計算
  • 利点: 工数エラーを月初に検出、月内に修正可能

    • ⏳ 損益Excel到着待ち: 経理チームから損益Excelが到達するまで待機

  • 実務では月によって異なる（15日～25日ごろ）
  • 決算スケジュール、年度末等の影響あり

    • ✅ Stage 2（15日～25日ごろ）: 経理から損益Excelが到着したら実施

  • 最新の損益データを反映したコスト按分
  • 経理システム用 CSV 生成（keiri_YYYYMM.csv）
  • 利点: 最新の原価データを反映、正確な按分が可能

    • メリット

    1. 早期フィードバック: Stage 1 で工数エラーを月初に検出・修正

    2. 柔軟な経理連携: 経理チームのスケジュール変更に対応

    3. 月内完結: 稼働5日＋損益Excel到着後で月内に完了

    4. エラー削減: 各段階で確認・修正が可能

    ⚠️ バッチファイル（.bat）の取り扱い注意（必読）

    run_process.bat などのバッチファイルは、画面に日本語のメッセージを表示します。

    以前は環境によって日本語が文字化けすることがありましたが、現在はすべてのバッチを正しい日本語（CP932）で保存し直したため、文字化けは発生しません。

    この状態を保つために、運用上の 重要なお願い があります。

  • ❌ .bat ファイルを直接メモ帳や他のエディタで開いて編集・保存しないでください。

    • エディタで保存し直すと文字コードが変わり、日本語が再び文字化けする原因になります。

  • ✅ バッチの内容を変更したい場合は、システム部門に依頼してください。

    • バッチは make_bat.py という専用ツールから正しい文字コードで再生成する仕組みになっています。修正が必要なときは、このツールで作り直します（集計担当者が手で直す必要はありません）。

    > もし誤って .bat を編集してしまい文字化けした場合も、システム部門が make_bat.py を実行すれば元の正しい状態に戻せます。慌てず連絡してください。

    第2章 月次作業スケジュール

    実務スケジュール（2段階処理）

    実務では、月初の早期段階と経理部門のスケジュール合わせて、2つのバッチを使い分けます。

    ステップ1: 稼働5日以内（工数集計～検証処理）

    タイミング: 月初から5営業日以内に実施

            
            実行スクリプト: run_process.bat

    実行日の目安（例：2026年4月の場合）:

            
            4月1日（水）～ 4月7日（火）の間で実施
            目安: 4月2日（木）または 4月3日（金）

    処理内容:

    1. 工数集計（aggregate_hours.py）

    2. 勤怠検証（verify_hours.py）

    3. 経理CSV出力（export_keiri_csv.py）

    4. 実績時間配分（allocate_actual_hours.py）

    5. 人月計算（calculate_person_months.py）

    所要時間: 約5分

    出力ファイル:

  • raw_data.csv - 工数集計結果
  • YYYYMM_actual_hours.csv - 実績配分結果

    • この時点での確認:

  • ✅ 工数エラー検出・修正（月初に対応可能）
  • ✅ 勤怠検証（異常値をチェック）

    • ステップ2: 損益Excel到着時（15日～25日ごろ）

    タイミング: 経理チームから 損益Excel が到着した時点

  • 実務では月によって異なる（15日～25日ごろ）
  • 確認方法: 経理チームに「損益Excel到着」の連絡を受ける
    •         
            実行スクリプト: run_allocate_costs.bat

    処理内容:

    1. 実績時間配分（allocate_actual_hours.py）再実行

    2. コスト按分（allocate_costs.py）← 最新の損益データを反映

    所要時間: 約2分

    出力ファイル:

  • YYYYMM_allocated_costs.csv - 按分詳細結果
  • keiri_YYYYMM.csv - 経理提出用 ⭐

    • 月によって異なるスケジュール例

    例1: 損益Excel が 20日に到着

            
            4月1日-7日:  run_process.bat 実行
            4月20日:     run_allocate_costs.bat 実行 → 経理へ提出

    例2: 損益Excel が 25日に到着

            
            4月1日-7日:  run_process.bat 実行
            4月25日:     run_allocate_costs.bat 実行 → 経理へ提出

    例3: 損益Excel が 15日に到着（早期）

            
            4月1日-7日:  run_process.bat 実行
            4月15日:     run_allocate_costs.bat 実行 → 経理へ提出

    例4: 損益Excel が 23日に到着

            
            4月1日-7日:  run_process.bat 実行
            4月23日:     run_allocate_costs.bat 実行 → 経理へ提出

    第3章 事前準備（集計処理前）

    Step 1: ファイルの準備確認

    ✅ Stage 1 実行前に確認

    1. 従業員の工数ファイル (member フォルダ)

  • 全従業員のCSVファイルが揃っているか
  • ファイル名形式: _W太郎.csv または 1_T太郎.csv
  • 月初までに最新データが入力されているか

    • 2. 勤怠データ (kintai フォルダ)

  • HTML形式の勤怠ファイルが置かれているか
  • ファイル名例: kintai_202604.html
  • 必須項目：従業員名、実働時間

    • ✅ Stage 2 実行前に確認

    3. 損益Excel (soneki フォルダ)

  • Excelファイル（XLSX形式）が置かれているか
  • ファイル名例: soneki_202604.xlsx
  • 重要: 経理チームから「損益Excel到着」通知があったか確認
  • シート名が月名（例：「4月」）になっているか

    • Step 2: 環境設定ファイル（env）の作成・確認

    環境設定ファイル（env）は、システムがデータの置き場所を知るための設定ファイルです。

    ファイル場所: bin フォルダ内（bin\env、拡張子なし）

    初回のみ：make_env.py で env を作成

    env は 手作業で作らず、make_env.py を使って作成します。

    手作業だと行のズレや文字化けが起きやすいため、専用ツールで正しく生成します。

    作成手順（初回のみ。一度作れば毎月は不要）:

    1. エクスプローラで bin フォルダを開く（例: C:\data\project\bin\）

    2. アドレスバーに cmd と入力して Enter（その場所でコマンド画面が開く）

    3. 次のように入力して Enter

            
               python make_env.py

    > 💡 python で動かない（「Microsoft Store が開く」「'python' は認識されません」等）場合は py に読み替えてください（例: py make_env.py）。

    > これは Microsoft Store 版のダミー python が原因のことがあります。以降このマニュアルに出てくる手入力の python ○○.py も同様に py ○○.py で実行できます。

    > なお、バッチ（run_process.bat / run_allocate_costs.bat / run_all.bat）は py／python を自動判定するため、この読み替えは不要です（バッチ実行時の手当ては要りません）。

    4. 各項目を順に聞かれます。何も入力せず Enter を押すと、表示されている既定値（[ ] の中身）がそのまま使われます。通常はすべて Enter で問題ありません。

            
               ======================================================
                 env(環境設定ファイル)生成 - Enterで既定値を採用
               ======================================================
               プロジェクトルート(絶対パス) [C:\data\project]: （Enter）
               bin サブディレクトリ [\bin]: （Enter）
               member サブディレクトリ [\member]: （Enter）
               kintai サブディレクトリ [\kintai]: （Enter）
               list サブディレクトリ [\list]: （Enter）
               soneki サブディレクトリ [\soneki]: （Enter）
               results サブディレクトリ [\results]: （Enter）
               1人日の標準労働時間(時間) [7.5]: （Enter）

    5. 最後に「生成完了: env (CP932, 8行)」と表示されれば成功です。

    よく使うオプション:

    やりたいこと 入力するコマンド
    対話入力で作る（推奨） python make_env.py
    ルートだけ指定して一括作成 python make_env.py --root C:\data\project
    既存の env を上書きし直す python make_env.py --force
    フォルダ未作成のまま作る python make_env.py --no-check

    2回目以降：env の確認だけ

    一度 env を作成済みであれば、毎月の作り直しは不要です。

    bin\env ファイルが存在することだけ確認してください。フォルダ構成（保存場所）を変えた場合のみ、python make_env.py --force で作り直します。

    Step 3: 案件マスタ（inc_list）の確認・再生成

    工数集計では、案件ごとの 内外製区分（N/G/K/y） と 品区 を list\<期>_inc_list.csv から読み込みます（列: inc_num, naigaikubun, hinku）。これは原価按分の 対象判定・配分に直結する重要なマスタ です。

    > ⚠️ 重要: <期>_inc_list.csv は 手で編集しないでください。

    > このCSVは インシデント管理表.xlsx から 自動生成される派生ファイル です。直接編集しても、次回の再生成で上書きされて消えます。

    マスタの大元は「インシデント管理表.xlsx」

  • 案件マスタの 正（真実） は、プロジェクトルート直下の インシデント管理表.xlsx です。
  • ファイル内の各 「◆XXX期」シート（例: ◆189期）が各事業年度の原本です。
  • このシートの ｲﾝｼﾃﾞﾝﾄ№ / ＮＧＫｙ / 品区 の3列から、<期>_inc_list.csv が作られます。

    • 案件の追加・変更があったときの手順

    案件マスタを変更する必要があるとき（新しい案件の追加、内外製区分の修正など）は、次の順で作業します。通常はシステム部門またはマスタ管理者が実施します。

    1. インシデント管理表.xlsx を Excel で開いて編集 する

    （該当する「◆XXX期」シートに行を追加・修正）

    2. bin フォルダでコマンド画面を開き、次を実行する

            
               cd bin
               python make_inc_list.py --env env
  • これで 全期分 の list\<期>_inc_list.csv がまとめて再生成されます。
  • 特定の期だけ作り直したい場合は --period 189 のように期番号を付けます。
    •         
                 python make_inc_list.py --env env --period 189

    3. 「出力完了: ...（N件, シート=◆XXX期）」と表示されれば成功です。

    エラーが出たとき（書き込まれずに中断します）

    インシデント管理表.xlsx に不備があると、make_inc_list.py は CSVを書き込まずに中断 し、問題箇所を一覧表示します。検出される主なエラーは次のとおりです。

  • №・ＮＧＫｙ・品区のいずれかが空欄（行番号付きで表示）
  • ＮＧＫｙが値域外（許容は N, G, K, y のみ）
  • 同じ№で内外製区分・品区が矛盾している（前回行と今回行の値が違う）

    • 表示された行番号を手がかりに xlsx 側を直してから、もう一度コマンドを実行 してください。エラーが残っている間は、既存の inc_list.csv は変更されません（安全に何度でも再実行できます）。

    > 📌 補足: インシデント管理表.xlsx に画像が貼られていても読み込めるように改善済みです（環境によっては以前うまく開けないことがありましたが、現在は問題ありません）。

    > 📌 補足（ダッシュボードの案件名）: 上記の インシデント管理表.xlsx からは、inc_list.csv とは別に list\inc_name_list.csv（インシデント番号→案件名, CP932） も派生生成され、配布物に同梱されます。ダッシュボードのデータ再生成（第11章, python bin\generate_dashboard_data.py）はこのCSVから案件名を読み込むため、インシデント管理表.xlsx を持たない配布版でも案件名込みで再生成できます。

    • 第7章 トラブルシューティング

    よくあるエラーと対応方法

    ❌ エラー1: 「ファイルが見つかりません」

    症状: 処理中に以下のメッセージが出て止まる

            
            エラー: raw_data.csv が見つかりません

    原因: 工数ファイルが member フォルダに入っていない、または Stage 1 が実行されていない

    対応方法:

    1. member フォルダをチェック

    2. _W従業員名.csv 形式のファイルがあるか確認

    3. Stage 1 が完了しているか確認

    4. もう一度実行

    ❌ エラー2: 「損益Excelファイルが読めません」

    症状:

            
            Excelファイル 'soneki.xlsx' が読み込めませんでした

    原因: 損益ファイル（Excel）が見つからない、または形式が違う

    対応方法:

    1. soneki フォルダをチェック

    2. .xlsx ファイルがあるか確認（.xls や .xlsm ではダメ）

    3. Excelで開いて、シート名が「4月」など月名になっているか確認

    4. 経理部門に「損益Excel到着」の連絡がされているか確認

    ❌ エラー3: 出力ファイルが空または 0 になっている

    症状:

            
            raw_data.csv は生成されたが、全部 0 になっている

    原因: 従業員の工数ファイルが見つからないか、形式が違う

    対応方法:

    1. member フォルダ内のCSVファイルを1つ開く

    2. Excel で見たときに、以下のような形式か確認

            
               日付,タスク名,工数
               44000,テスト_001,8
               44001,テスト_002,8

    3. ファイル名が _W で始まるか、または 数字_T で始まるか確認

    4. 形式が違う場合は、担当者に修正を依頼

    ⚠️ 警告: 「member CSV をエンコード不正のためスキップしました」

    症状: Stage 1（工数集計）の途中で、次のような警告がまとめて表示される

            
            【警告】以下の member CSV はエンコード不正のためスキップしました（集計から除外）
              - _W太郎.csv

    原因: その従業員の工数ファイルが CP932（Shift-JIS）以外（例: UTF-8）で保存されている

    以前との違い:

  • 以前は、1つでもエンコードの違うファイルがあると 文字化けで処理全体が止まって いました。
  • 現在は、該当ファイルだけを除外し、ファイル名を警告に出して処理を継続 します。残りの従業員は問題なく集計されます。

    • 対応方法:

    1. 警告に表示されたファイルを、保存し直してもらう（文字コードを Shift-JIS / CP932 に）

  • Excel から「CSV（コンマ区切り）」で保存すれば通常 CP932 になります
  • メモ帳の場合は「名前を付けて保存」で文字コードを ANSI にする

    • 2. 直したファイルを member フォルダに置き直し、Stage 1 を再実行

    3. スキップされた従業員は集計に 含まれていない ため、必ず直して入れ直すこと

    > 🔎 検収・チェックを厳密にしたいとき: 担当者がコマンドで実行する場合、aggregate_hours.py に --strict を付けると、エンコード不正ファイルを スキップせずエラーで停止 します（見落とし防止用）。通常運用のバッチは従来どおりスキップ＋警告で継続します。

    ⚠️ 警告: 「工数が過大／過小です（単位をご確認ください）」

    症状: Stage 1 で、次のような工数の 単位（分／時間）の取り違い を疑う警告が出る

            
            【警告】単位の取り違いが疑われる入力があります:
              ...（日合計が 24h を超える、または月合計が 1h 未満 等）

    原因: 工数は 分単位 で入力する想定ですが、誤って 「時間」で入力（または分のつもりが桁違い）している可能性があります。

  • 日合計が 24 時間を超える → 過大入力の疑い
  • 月合計が 1 時間未満（かつ 0 でない） → 「時間」を分として扱い縮小された疑い

    • 重要: この警告は 出力をブロックしません（処理は最後まで進みます）。あくまで気づきのための注意喚起です。

    対応方法:

    1. 警告に出た従業員・日付の入力値を確認

    2. 単位の取り違いがあれば修正してもらい、Stage 1 を再実行

    3. 妥当な値（深夜作業等で実際に大きい等）であればそのまま進めてよい

    ❌ エラー4: 「環境設定ファイルが見つかりません」

    症状:

            
            設定ファイル 'project.env' が見つかりません

    原因: env ファイルが無い、またはパスが違う

    対応方法:

    1. bin フォルダをチェック

    2. env というファイル（拡張子なし）があるか確認

    3. ない場合は、bin フォルダで python make_env.py を実行して作成する

    （作成手順は「第3章 Step 2: 環境設定ファイル（env）の作成・確認」を参照）

    4. 既に env があるのにエラーが出る場合は、python make_env.py --force で作り直す

    ❌ エラー5: 「'python' は内部コマンドまたは外部コマンドとして認識されていません」

    症状: バッチファイルを起動した直後に上記メッセージが出て終了する

    原因: PC に python コマンドが PATH 登録されていない（py ランチャーのみの環境）

    対応方法:

  • run_process.bat / run_allocate_costs.bat / run_all.bat は python が見つからない場合に 自動的に py コマンドで再試行 します。通常はそのまま処理が続きます。
  • それでも失敗する場合は、Python 公式サイトから Python をインストールし、「Add Python to PATH」にチェックを入れて再インストールしてください。

    • ⚠️ 警告: 「工数の合計が実働時間と異なります」

    症状:

            
            verify_YYYYMM.csv に警告が記録されている

    原因: 従業員の入力工数と、勤怠システムの実働時間に差がある

    対応方法:

  • 軽度の差異（±1時間以内）: 許容範囲 → そのまま進める
  • 大きな差異（±5時間以上）: 当人に確認 → 修正後に再実行
  • 判断できない場合は、管理者に報告

    • 付録 FAQ（よくある質問）

    Q1: Stage 1 と Stage 2 をまとめて実行できませんか？

    A: いいえ。損益Excel待ちの期間があるため、2段階に分ける必要があります。

    Q2: 損益Excel がいつ来るかわかりません

    A: 経理チームに「損益Excel予定日」を事前に確認することをお勧めします。

    Q3: 毎月同じ手順ですか？

    A: はい。月号を変えるだけで、手順は同じです。

    Q4: Stage 1 で工数エラーが見つかりました。どうしましょう？

    A: 入力者に修正を依頼します。Stage 1 を月初に実行することで、月内に対応できます。

    Q5: 複数月をまとめて処理できますか？

    A: 1ヶ月ずつの実行を推奨します。

    Q6: データのバックアップは？

    A: results/ と keiri/ フォルダを定期的にコピーしておくことを推奨。
