# 工数集計・按分システム（manhours） 要件定義書

**作成日**: 2026-05-21　**最終更新**: 2026-06-19  
**プロジェクト名**: 工数集計・按分システム（manhours）  
**バージョン**: 1.1

---

## 1. プロジェクト概要

工数集計・按分システムは、従業員の工数（作業時間）、勤怠データ、原価データを統合的に管理し、プロジェクトごとに工数を集計・按分して経理データを生成するシステムです。

### 背景・目的
- 複数のプロジェクトに従事する従業員の工数を正確に把握
- 原価データに基づいて工数をプロジェクト・案件ごとに按分
- 経理システムへの入力データを自動生成し業務効率化
- 工数と実績時間の整合性を検証

---

## 2. スコープ

### 対象ユーザー
- プロジェクト管理者
- 経理部門
- 工数管理者

### 処理範囲
工数データの入力から経理データの出力までの全プロセスをカバー：

```
工数入力 → 工数集計 → 勤怠検証 → 実績時間配分 → コスト按分 → 人月計算 → 経理CSV出力
```

### 対象外
- 従業員の工数入力UI（Excelテンプレート等の提供のみ）
- 給与計算システムとの連携
- プロジェクト管理システムとの自動連携

### 入力データ形式

#### member ディレクトリ（工数入力）
- ファイル名形式: `_W従業員名.csv` または `序号_T従業員名.csv`
- 内容: 日付、タスク名、工数（時間）

#### list ディレクトリ（マスターデータ）
- `*_list.csv` - プロジェクト/タスク/品区マスタ
- `hinku_list.csv` - 内外製区分コード定義
- `<期>_inc_list.csv` - 案件マスタ（列: `inc_num, naigaikubun, hinku`）。**`make_inc_list.py` が `インシデント管理表.xlsx` から自動生成する派生ファイル**。手編集禁止（詳細は「3.1 案件マスタのアーキテクチャ」参照）。

#### kintai ディレクトリ（勤怠データ）
- HTML形式またはXLS偽装HTML
- 内容: 従業員名、雇用区分、実働時間

#### soneki ディレクトリ（原価データ）
- Excel形式（XLSX）
- 内容: 月別の労務費・経費データ

### 出力データ形式

#### results ディレクトリ
- `raw_data.csv` - 集計後の工数データ（月・案件・品区・従業員）
- `集計_YYYYMM.csv` - 月別集計結果

#### keiri ディレクトリ
- `YYYYMM_allocated_costs.csv` - 按分後の経理データ
- `keiri_YYYYMM.csv` - 経理出力形式

### 3.1 案件マスタのアーキテクチャ（一元管理）

案件ごとの **内外製区分（N/G/K/y）** と **品区** は、原価按分の対象判定・配分に直結する重要なマスタである。この情報は **単一の原本で一元管理** し、派生ファイルを自動生成する方式を採用する。

```
[原本] インシデント管理表.xlsx
        各「◆XXX期」シート（ｲﾝｼﾃﾞﾝﾄ№ / ＮＧＫｙ / 品区 の3列）
              ↓  make_inc_list.py で自動生成
[派生] list/<期>_inc_list.csv
        列: inc_num, naigaikubun, hinku（CP932）
              ↓  aggregate_hours.py 等が読み込む
[利用] 工数集計・原価按分の対象判定／配分
```

**原則**:
- 案件マスタの **真実（正）は `インシデント管理表.xlsx`**。各「◆XXX期」シートが各事業年度の原本。
- `<期>_inc_list.csv` は **`make_inc_list.py` が xlsx から生成する派生物であり、手編集しない**（再生成で上書きされる）。

**`make_inc_list.py` の仕様**:
- 入力: `インシデント管理表.xlsx`（プロジェクトルート直下、`--xlsx` で上書き可）
- 出力: `list/<期>_inc_list.csv`（CP932、ヘッダー `inc_num,naigaikubun,hinku`）
- パラメータ:
  ```
  --env ENV_FILE    環境設定ファイルパス（既定: env）
  --period N        対象の期番号（例: 189）。未指定なら ◆XXX期 シートを全件処理
  --xlsx PATH       インシデント管理表.xlsx のパス（未指定ならルート直下）
  ```
- **バリデーション（不備があれば一覧表示してCSVを書き込まず中断）**:
  - №・ＮＧＫｙ・品区のいずれかが空欄
  - ＮＧＫｙ が値域外（許容: `N`, `G`, `K`, `y`）
  - 同一№で内外製区分・品区が矛盾（前回行と値が異なる）
- **堅牢化**: `read_only=True` で読み込み、xlsx に埋め込み画像があっても新しい openpyxl でクラッシュせず読めるようにした。
- 全シートを先に走査してエラーが無いことを確認してから書き込むため、エラーが残る間は既存の `inc_list.csv` を破壊しない（安全に再実行可能）。

> 注: この再生成は通常システム部門／マスタ管理者の作業。HTML入力UI・インシデント管理表のHTMLエディタは **本書の対象外**（将来の任意ツール）。本書は従来運用（xlsx を正としたマスタ管理）を前提とする。

## 4. 主要機能

### 4.1 工数集計（aggregate_hours.py）

**目的**: 従業員の個別工数データから、プロジェクト・案件ごとの工数を集計

**入力**: 
- member ディレクトリの従業員別CSV
- master_csv（プロジェクト・タスクマスタ）

**処理**:
1. 従業員CSVを読み込み、日付をExcel形式から YYYY/MM に変換
2. タスク名からプロジェクト・品区を特定
3. 案件ごとに従業員別の工数を集計
4. 「内外製区分」により対象/非対象を判定（N/G/K は対象、y で品区チェック）

**出力**: `raw_data.csv`

**パラメータ**:
```
--env ENV_FILE    環境設定ファイルパス
--month YYYYMMDD  対象月（形式: YYYYMMDD）
--strict          エンコード不正の member CSV をスキップせずエラー終了する（検収用）
```

**エラーハンドリング**:
- envファイル未検出
- 入力ファイル未検出
- 形式エラー（日付、数値）

**堅牢化（後方互換・既定動作は不変）**:
- **エンコード不正 member CSV のスキップ**: member CSV が UTF-8 等で誤保存され読み込めない場合、従来は文字化けで処理全体がクラッシュしていた。現在は **該当ファイルだけを集計から除外し、ファイル名付きの警告を出して処理を継続** する。`--strict` 指定時のみ、スキップせずエラー終了する（見落とし防止の検収モード）。
- **単位サニティ警告**: 工数の単位取り違い（分でなく時間で入力）の兆候を検知して警告する。**日合計 > 24h（=1440分）** で過大入力、**月合計 < 1h（かつ > 0）** で縮小入力を疑う。この警告は **出力をブロックしない**（`--strict` でも警告に留める）。

---

### 4.2 勤怠検証（verify_hours.py）

**目的**: 工数入力と勤怠データの整合性を検証

**入力**:
- raw_data.csv（aggregate_hours の出力）
- kintai ディレクトリの勤怠HTML

**処理**:
1. 勤怠HTMLから従業員名・雇用区分・実働時間を抽出
2. 工数合計と実働時間の差異を検出
3. 警告/エラー情報をレポート

**出力**: 
- `verify_YYYYMM.csv` - 検証結果
- コンソールに警告メッセージ

**パラメータ**:
```
--env ENV_FILE    環境設定ファイルパス
--month YYYY/MM   対象月（形式: YYYY/MM）
```

---

### 4.3 実績時間配分（allocate_actual_hours.py）

**目的**: 工数を実際の勤務時間に基づいて配分

**入力**:
- raw_data.csv
- kintai データ（実働時間）

**処理**:
1. 従業員の実働時間を取得
2. 工数合計に対する実働時間の比率を計算
3. 各案件の工数に配分率を適用

**出力**: `YYYYMM_actual_hours.csv`

**パラメータ**:
```
--env ENV_FILE        環境設定ファイルパス
--month YYYY/MM       対象月
--standard-hours N    標準勤務時間（デフォルト: envファイルから取得）
```

---

### 4.4 コスト按分（allocate_costs.py）

**目的**: 原価データに基づいて工数あたりのコストを計算・按分

**入力**:
- raw_data.csv
- soneki Excel（月別労務費・経費）

**処理**:
1. 対象月のsoneki Excelから労務費・経費を取得
2. 工数ベースで案件ごとにコストを按分
3. 工数単価（時間あたりコスト）を計算

**出力**: `YYYYMM_allocated_costs.csv`

**パラメータ**:
```
--env ENV_FILE    環境設定ファイルパス
--month YYYYMM    対象月（形式: YYYYMM）
--data CSV_FILE   raw_data.csv パス（オプション）
```

**対象判定ロジック**:
- `naigai='N'`: 常に対象
- `naigai='G'`: 常に対象
- `naigai='K'`: 常に対象
- `naigai='y'`: 品区が y1, y10, y20 の場合のみ対象

---

### 4.5 人月計算（calculate_person_months.py）

**目的**: 従業員ごとの人月を計算（後続の人員計画・給与計算向け）

**入力**: 実績時間配分データ

**処理**:
1. 従業員ごとの総工数を集計
2. 標準勤務時間で除算して人月を計算
3. 案件別・従業員別クロス集計

**出力**: `YYYYMM_person_months.csv`

---

### 4.6 経理CSV出力（export_keiri_csv.py）

**目的**: 経理システム向けのCSVを生成

**入力**: 
- コスト按分結果
- 人月計算結果

**処理**:
1. 案件・品区・従業員ごとに売上原価を計算
2. 経理フォーマットに整形
3. 月次決算データとして出力

**出力**: `keiri_YYYYMM.csv`

---

### 4.7 人員情報整理（rearrange_person.py）

**目的**: 従業員情報の統一・整合性確保

**入力**: member ディレクトリの従業員CSV

**処理**:
1. ファイル名から従業員名を抽出
2. 重複・異表記を検出
3. マスタの従業員リストと照合

**出力**: `person_info.csv`

---

### 4.8 データ更新（update_data.py）

**目的**: 月次処理時の中間データを更新

**処理**:
1. 前月データのアーカイブ
2. 当月データディレクトリの初期化
3. マスタデータの整合性チェック

---

### 4.9 案件マスタ生成（make_inc_list.py）

**目的**: 案件マスタの原本 `インシデント管理表.xlsx` から、各スクリプトが読み込む `<期>_inc_list.csv` を自動生成する

**入力**: `インシデント管理表.xlsx`（各「◆XXX期」シート）

**出力**: `list/<期>_inc_list.csv`（CP932、列: `inc_num, naigaikubun, hinku`）

詳細な仕様・バリデーション・堅牢化は「3.1 案件マスタのアーキテクチャ」を参照。

---

### 4.10 バッチファイル生成

**目的**: 手動実行用のバッチファイルを自動生成

**スクリプト**:
- `gen_bat.py` - 基本バッチテンプレート生成
- `make_bat.py` - 実行用バッチスクリプト作成
- `make_run_process.py` - フェーズごとのバッチ生成
- `make_run_all.py` - 全処理の一括実行バッチ生成

**用途**: 経理部門が管理画面なしに月次処理を実行するためのツール

## 5. データフロー

```
[1] 工数入力（member/*.csv）
           ↓
[2] aggregate_hours.py → raw_data.csv
           ↓
[3] verify_hours.py → verify_YYYYMM.csv（検証結果）
           ↓
[4] allocate_actual_hours.py → YYYYMM_actual_hours.csv
           ↓
[5] allocate_costs.py + soneki/*.xlsx → YYYYMM_allocated_costs.csv
           ↓
[6] calculate_person_months.py → YYYYMM_person_months.csv
           ↓
[7] export_keiri_csv.py → keiri_YYYYMM.csv
           ↓
[8] keiri/ へ出力
```

## 6. 非機能要件

### 6.1 パフォーマンス
- 月間工数処理時間: 5 秒以内（通常運用）
- 従業員数: 100名までスケール対応

### 6.2 信頼性
- 全スクリプトの単体テストカバレッジ: 70% 以上
- 統合テスト: 主要処理フロー全て検証
- エラーハンドリング: 入出力エラーに対応

### 6.3 保守性
- コード自動テスト（pytest）による品質保証
- 月次処理の手動介入最小化
- 自動バッチによる定期実行対応

### 6.4 データセキュリティ
- 入出力ファイルの文字コード: CP932（Windows日本語対応）
- 個人情報（従業員名）を含むため、アクセス制限推奨

- **Python**: 3.7 以上

| 形式 | 文字コード | 用途 |
|------|----------|------|

- **内部**: YYYY/MM（例: 2026/04）

| コード | 意味 | 対象判定 |

## 10. 用語集

| 用語 | 説明 |
|------|------|
| 工数 | 案件に従事した時間（時間単位） |
| 人月 | 標準勤務時間（例8時間）で正規化した工数（160時間 ≈ 1人月） |
